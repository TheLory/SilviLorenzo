Introduction

there is no code without a project, no project without problem analysis and no problem without requirements

Requirements

Design and build a software system that leads a robot to walk along the boundary of a empty, rectangular room.

Requirement analysis

The discussion with the customer clarified that he give to the nouns the following meanings:
  • room: a convetional room that can be found in all normal building
  • boundary: the perimeter of the room, which in a normal room is bordered by (walls)
  • robot: a device able to move reciving inputs using the web, we use the rules expressed in VirtualRobot2021.html.
Reguarding to the action expressed (verbs):
  • walk: the robot must be capable of moving forward, alongside the room's walls

First user story

As an user, i put the robot in the home cell south facing, than i start a sistem which send to the robot the movemenets commands it must follow (wifi network) also the robot can't be stopped: the system must end the process independently, once the task is completed.

A the end of the system execution, i expect that the robot has completed the track (only once) as reported in the side image

Verify of the expected result (Test plans)

Must be verified that the path done by the robot is the expected one
the verification of the path must be done using a software, without human intervention

Problem analysis

Relevant aspects

  1. it regard building a distribuited system made up of two macro-components
    • the virtual robot given by the customer
    • our application (boundaryWalk) which send commands to the robot with the aim of satisfy the requirements
  2. the robot can be driven via network in two different ways, has explained in VirtualRobot2021.html: commands:
    • sending message to the door 8090 using HTTP POST protocol
    • sending message to the door 8091 using a websocket
  3. considering that a lot of libraries exist in different programming language which provide to send messagges, no one abstraction-gap was found on the operative side.
    however an abstraction gap on concenptual side exist: there are two different ways to send message to the robot and each one needs appropriate support . nevertheless it is all about build up a request-response interaction schema .
  4. it is estimated that a first prototype of the application could be ready at least in tree days
